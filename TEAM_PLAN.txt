## 2. THE BEST STRATEGY & ARCHITECTURE PLAN

### A. The Backend Framework (FastAPI)
We will use **FastAPI** instead of Flask. Since our OCR inference is CPU-heavy and blocks the thread, we will use `run_in_executor()` to offload the ML processing to a thread pool so the server stays responsive for other requests.

### B. The Database Strategy (SQLite -> PostgreSQL)
We will use **SQLite** for Phase 1 while we build the app. It requires zero installation and saves as a single local file. In Phase 2, we will switch to **PostgreSQL**. Because we will use SQLAlchemy as our ORM, the switch only requires changing the database connection string.

### C. The ML Parallel Development Strategy
We will use the **pre-built TrOCR** model (`microsoft/trocr-base-handwritten`) immediately. This model reads clean handwriting well, which is perfect for testing the full app pipeline. Person 1 will train a custom model on real doctor prescription images using Google Colab in the background. When the custom model is ready, we swap the model path in the config — no other code changes needed.

### D. The Testing Strategy
For Phase 1 testing, we will write prescriptions in **clean, neat handwriting** on paper, photograph them, and use those images to test the full pipeline. This lets us verify that Backend → OCR → Extraction → Drug Check → Frontend all work correctly, without fighting messy handwriting accuracy issues.

## 5. THE API CONTRACT (CRITICAL — EVERYONE MUST USE THIS)

This is the exact JSON format that the backend returns. Person 2 builds the API to return this. Person 3 builds the frontend to consume this. No deviations.

### `POST /api/upload`
**Request:** `multipart/form-data` with field `file` containing the prescription image.

**Response (200 OK):**
```json
{
  "success": true,
  "raw_text": "Amoxicillin 500mg\nIbuprofen 200mg\nTake twice daily",
  "medicines": [
    {
      "name": "Amoxicillin",
      "dosage": "500mg",
      "frequency": "twice daily"
    },
    {
      "name": "Ibuprofen",
      "dosage": "200mg",
      "frequency": "twice daily"
    }
  ],
  "warnings": [
    {
      "drug_1": "Amoxicillin",
      "drug_2": "Ibuprofen",
      "severity": "moderate",
      "message": "May reduce effectiveness of Amoxicillin. Monitor closely."
    }
  ],
  "prescription_id": 1
}
```

**Response (error):**
```json
{
  "success": false,
  "error": "Could not read text from the image. Please upload a clearer image."
}
```

### `GET /api/history`
**Response:**
```json
{
  "prescriptions": [
    {
      "id": 1,
      "date": "2026-04-22T18:30:00",
      "raw_text": "Amoxicillin 500mg...",
      "medicine_count": 2,
      "warning_count": 1
    }
  ]
}
```

## 7. TECHNICAL GUIDES: HOW TO DO THE WORK

### A. How to add the pre-built OCR model for testing (For Person 2)
1. Open `ocr/ocr_engine.py`. This script already downloads and runs a pre-trained model: `microsoft/trocr-base-handwritten`.
2. **IMPORTANT:** Add an empty `__init__.py` file inside the `ocr/` folder. Without this, Python cannot import it as a package.
3. In your FastAPI `main.py`, import it: `from ocr.ocr_engine import extract_text`.
4. **IMPORTANT:** The model takes 30-60 seconds to load on first import. Use lazy loading — load the model only when the first request comes in, not at server startup. This makes the server start instantly.
5. Since OCR is CPU-heavy and blocks the thread, wrap the call in `run_in_executor()` so the server stays responsive:
   ```python
   import asyncio
   loop = asyncio.get_event_loop()
   result = await loop.run_in_executor(None, extract_text, image_path)
   ```
6. Create a `/api/upload` route that accepts an image file via `multipart/form-data`.
7. Save the uploaded image to a temp folder, pass its path to `extract_text()`, return the response matching the API contract above.
8. Clean up temp files after processing.

### B. How to make the database (For Person 2)
1. Install: `pip install sqlalchemy`
2. Create `database/db.py` with SQLAlchemy setup using SQLite.
3. **IMPORTANT:** Add an empty `__init__.py` file inside the `database/` folder.
4. Create these tables:
   - `Users`: id, name, email, password_hash
   - `Prescriptions`: id, user_id, image_url, raw_text, structured_medicines (JSON), interaction_warnings (JSON), status, created_at
   - `DrugInteractions`: id, drug_1, drug_2, severity, warning_message

### C. How to build extraction & drug checker (For Person 3 — NO backend dependency)
1. **You can start immediately.** Use the sample OCR outputs in Section 6 above as test inputs.
2. Build `medicine/extractor.py` — a Python function that takes raw OCR text and returns structured medicine JSON.
3. Build `drug_interaction/checker.py` — a Python function that takes a list of medicines and checks them against a CSV of known interactions.
4. Build `drug_interaction/interactions.csv` — start with 50 common harmful drug pairs.
5. **IMPORTANT:** Add empty `__init__.py` files inside `medicine/` and `drug_interaction/` folders.
6. Test your functions independently: `python -c "from medicine.extractor import extract_medicines; print(extract_medicines('Amoxicillin 500mg'))"`.
7. When Person 2's backend is ready (Day 3), they will import your functions into the API endpoint.

### D. How to connect the frontend to backend (For Person 3 — Day 3)
1. In the React Native app, open `src/screens/UploadScreen.js`.
2. Replace the `setTimeout` mock with a real `fetch` call using `FormData`.
3. The response JSON matches the contract in Section 5 — use those exact field names.
4. Pass the response to `PrescriptionDetailsScreen.js` via `navigation.navigate('PrescriptionDetails', { data: response })`.
5. Rewrite `PrescriptionDetailsScreen.js` to render `data.medicines` and `data.warnings` dynamically instead of hardcoded values.
6. **Android emulator:** Use `10.0.2.2:8000` instead of `localhost`. **Physical device:** Use your PC's LAN IP (e.g., `192.168.1.X:8000`).
7. **DO NOT** set `Content-Type` header manually with FormData — `fetch` sets it automatically with the correct boundary.
